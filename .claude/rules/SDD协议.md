# SDD (Software Design Document) Protocol

## 目的

确保所有代码实现都有明确的设计文档支撑，避免边做边改、架构腐化。

## 核心原则

1. **设计先行**：代码实现前必须先完成 SDD
2. **审核批准**：SDD 必须经过审核才能实施
3. **按图施工**：实现必须严格遵循已批准的 SDD
4. **变更追溯**：设计变更必须更新 SDD 并记录原因

## SDD 模板

```markdown
# SDD_<项目>_<版本>_<日期>

## 1. 概述

### 1.1 目的
[简述此功能/模块的目的]

### 1.2 范围
[功能边界，与其他模块的关系]

### 1.3 定义
- Term1: 定义
- Term2: 定义

### 1.4 参考
- 相关文档/设计
- 技术约束

## 2. 系统架构

### 2.1 组件图
[使用文本图或 Mermaid]

### 2.2 模块关系
[模块间依赖]

## 3. 接口定义

### 3.1 API 接口
| 接口 | 方法 | 输入 | 输出 | 说明 |
|------|------|------|------|------|
| /api/xxx | POST | JSON | JSON | 说明 |

### 3.2 示例
```json
// 请求
{}
// 响应
{}
```

## 4. 数据结构

### 4.1 数据模型
```python
@dataclass
class Xxx:
    field: type  # 说明
```

### 4.2 数据库 Schema（如适用）
```sql
CREATE TABLE ...
```

## 5. 行为设计

### 5.1 状态机
[状态转换图]

### 5.2 流程图
```
开始 → 步骤1 → 步骤2 → 结束
```

## 6. 非功能性需求

### 6.1 性能
- 响应时间 < Xms
- 吞吐量 > Y req/s

### 6.2 安全
- 认证/授权要求
- 数据加密

### 6.3 可靠性
- 可用性目标
- 错误处理

## 7. 变更记录

| 日期 | 版本 | 变更内容 | 作者 |
|------|------|---------|------|
| 2024-01-01 | 0.1 | 初始版本 | xxx |
```

## 文档结构

1. **概述** - 目的、范围、定义
2. **系统架构** - 组件图、模块关系
3. **接口定义** - API、消息格式、示例
4. **数据结构** - 数据模型、数据库 schema
5. **行为设计** - 状态机、流程图
6. **非功能性需求** - 性能、安全、可靠性
7. **变更记录** - 版本历史

## 编写要求

- 明确标注版本号和日期
- 所有接口需提供示例（请求/响应 JSON）
- 变更记录在文档末尾
- 代码实现前必须先完成 SDD

## 审阅流程

```
草稿编写 → 同行评审 → 架构师审核 → 批准发布
   ↓            ↓           ↓
  修订       修订      实施/再审
```

| 阶段 | 负责 | 说明 |
|------|------|------|
| 草稿编写 | 作者 | 编写完整 SDD |
| 同行评审 | 同事 | 检查技术正确性 |
| 架构师审核 | 架构师 | 检查设计合理性 |
| 批准发布 | 负责人 | 最终批准 |

## 文件命名

`SDD_<项目>_<版本>_<日期>.md`

示例：`SDD_user_auth_1.0_20240101.md`

## 强制执行

1. **创建文件前**：检查是否存在相关 SDD
2. **开始编码前**：确认 SDD 已批准
3. **代码审查时**：检查是否符合 SDD 设计
4. **变更发生时**：更新 SDD 再改代码

## SDD 状态

| 状态 | 说明 | 可开始编码 |
|------|------|-----------|
| Draft | 草稿 | ❌ |
| In Review | 评审中 | ❌ |
| Approved | 已批准 | ✅ |
| Deprecated | 已废弃 | ❌ |

## 示例流程

```
1. 用户请求：需要实现用户登录功能

2. Agent 响应：
   "我将先编写 SDD，然后再实现代码。"
   → 创建 SDD_user_login_0.1_20240101.md

3. SDD 内容：
   - 概述：登录功能，包含用户名密码验证
   - 接口：POST /api/login
   - 数据结构：User, LoginRequest, LoginResponse
   - 行为：状态机描述登录流程
   - 安全：密码加密要求

4. 审核通过后：
   → 开始按 SDD 实现代码

5. 实现完成：
   → 验证符合 SDD 设计
```

## 快速检查清单

- [ ] 目的和范围清晰
- [ ] 接口定义完整（含示例）
- [ ] 数据结构明确定义
- [ ] 行为设计包含状态机/流程图
- [ ] 非功能性需求列出
- [ ] 变更记录模板已准备好
